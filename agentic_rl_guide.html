Overview

一张总览图

先把 Agentic RL 当成一张系统图，而不是一堆缩写。它关注的核心对象不是单轮回答，而是多步轨迹； 它优化的不只是“回答像不像人喜欢的样子”，而是“能不能在复杂环境里完成任务”。

Agentic RL 的核心对象

不是单轮回复，而是多步轨迹。模型需要在观察、思考、调用工具、执行动作、接收反馈之间循环前进。

为什么 2024 之后明显升温

因为单纯的 SFT 很难让模型在长链推理、复杂搜索和真实环境交互里持续优化，RL 提供了“按结果回传信用”的机制。

和传统 RLHF 的区别

传统 RLHF 更像对回答风格与偏好的微调；Agentic RL 更强调过程质量、工具使用、任务成功率和多步决策稳定性。

Momentum

为什么 Agentic RL 会爆发

可以把近几年的路径看成两条线逐渐汇合。第一条线是 test-time agent，第二条线是 training-time RL。 前者告诉我们“推理时可以像 agent 一样搜索和行动”，后者告诉我们“这些行为模式可以被训练进去，而不只是靠 prompt 临时诱导”。

早期 RLHF
重点是对齐回答风格、帮助性与安全性。单位样本通常是一轮回答，奖励更偏向偏好模型或人工反馈。
ReAct / ToT
让模型在推理时显式地产生“思考 + 行动”结构，证明多步推理和搜索可以显著改善复杂任务表现。
SWE-agent
模型开始进入真实软件环境，面对文件系统、测试反馈、命令执行和长序列决策，agent 训练需求迅速清晰。
DeepSeek-R1 之后
社区开始集中关注“推理能力能否被 RL 系统性放大”，于是 GRPO、DAPO、GSPO、verifier-driven RL 等路线快速升温。
从提示工程走向策略学习 从单轮回答走向多步轨迹 从偏好对齐走向任务求解
Math

数学基础：先把主干公式吃透

无论名字叫 PPO、GRPO、GSPO 还是 DAPO，底层都离不开策略优化。很多差异看起来像新缩写， 但本质上通常是在“优势怎么构造、ratio 怎么裁剪、KL 怎么限制、序列怎么聚合”上做变化。

令轨迹为 $\tau = (s_1, a_1, r_1, \dots, s_T, a_T, r_T)$，策略为 $\pi_\theta(a_t \mid s_t)$， 我们希望最大化总期望回报：

$$J(\theta) = \mathbb{E}_{\tau \sim \pi_\theta}[R(\tau)]$$
这里的 $R(\tau)$ 可以是最终答案是否正确，也可以是过程奖励、格式奖励、工具调用奖励、代码测试通过率等组合。

经典策略梯度写成：

$$\nabla_\theta J(\theta) = \mathbb{E}_{\tau \sim \pi_\theta}\left[\sum_{t=1}^{T} \nabla_\theta \log \pi_\theta(a_t \mid s_t) \cdot A_t\right]$$
$A_t$ 是优势函数，表示这个动作比“平均水平”好多少。大模型 RL 里，如何构造 $A_t$，就是很多算法差异的来源。

如果使用 value baseline，常见定义是：

$$A_t = Q(s_t, a_t) - V(s_t)$$

为了避免策略更新过猛，现代方法几乎都会加入参考策略约束，最常见的是 KL 正则：

$$\mathcal{L} = \mathbb{E}\left[\log \pi_\theta(a_t \mid s_t)\cdot A_t\right] - \beta \, \mathrm{KL}\left(\pi_\theta \parallel \pi_{\mathrm{ref}}\right)$$

PPO 的代表性形式是裁剪比率：

$$r_t(\theta) = \frac{\pi_\theta(a_t \mid s_t)}{\pi_{\theta_{\mathrm{old}}}(a_t \mid s_t)}$$ $$\mathcal{L}_{\mathrm{PPO}} = \mathbb{E}\left[\min\left(r_t(\theta) A_t, \mathrm{clip}(r_t(\theta), 1-\epsilon, 1+\epsilon) A_t\right)\right]$$
裁剪的意义是：奖励好的方向可以继续推，但不能一下推得太离谱，否则训练很容易炸掉。

组相对优势是 reasoning RL 常见套路，一个常见写法是：

$$A_i = \frac{r_i - \bar{r}}{\sigma(r_1, \dots, r_G) + \epsilon}$$
其中 $G$ 是组大小，$\bar{r}$ 是组内平均奖励。这个想法非常适合没有稳定 value model、但能拿到相对好坏信号的场景。

GSPO 这类序列级方法更强调整条回答的概率比率，直觉上是在把 token-level 更新改成 sequence-level 更新，从而让序列级奖励和优化粒度更一致。

Taxonomy

算法分类：别被缩写吓到，先看它改了哪一层

大模型 RL 算法很多，但你可以从四个问题去分类：奖励怎么来、优势怎么估、更新怎么裁剪、采样怎么组织。 这样看，PPO、RLOO、GRPO、GSPO、DAPO 就不会是散乱的缩写，而是一族可比较的方法。

方法族 核心想法 优点 常见痛点
PPO / PPO-style 显式 ratio clipping，常配 value model 与 KL penalty 框架成熟，理论和工程经验丰富 value 训练重，长序列成本高，调参复杂
RLOO / REINFORCE-family 弱化 value 依赖，用 leave-one-out 或 baseline 降低方差 实现更轻，适合序列级奖励 方差仍可能较大，对采样质量敏感
GRPO-family 同 prompt 采样一组回答，利用组内相对奖励做优势归一化 不强依赖 value model，适合 reasoning 场景 对组大小、奖励尺度、长度偏差敏感
GSPO 使用 sequence-level importance ratio 与 sequence-level clipping 更贴近序列级奖励，训练 MoE reasoning 模型更稳 细粒度 credit assignment 会变弱
DAPO 强调 decoupled clip、dynamic sampling 与大规模 RL 系统工程 更关注真实大规模训练可复现性 算法与系统细节强耦合，必须结合实现理解
DPO / ORPO / KTO 直接从偏好对构造优化目标，不显式 rollout RL 训练简单、稳定，适合偏好数据 对真实环境回报和长程 credit assignment 支持有限
PPO 是“标准教科书”

如果你想先搭一个可解释的 RL 训练闭环，PPO 很适合入门。它让你看清 ratio、clip、value、GAE、KL 等核心部件怎么协同工作。

GRPO 很适合 reasoning-style LLM

很多推理任务不容易给出精准 token-level value，但很容易判断多个候选答案谁更好，于是组相对优势就很自然。

GSPO 代表“序列粒度优化”

它把更新单位往整条回答上抬，试图让“序列级奖励”与“策略更新粒度”对齐，减少 token 粒度和 sequence 粒度之间的错位。

DAPO 更像工程增强包

它不是推翻策略梯度，而是针对长度偏置、裁剪耦合、采样效率、RL 系统可扩展性这些现实问题做强化。

如何理解 GRPO

同一个问题采样多条回答，奖励高的回答相对抬高概率，奖励低的回答相对压低概率。重点是“相对排序”而不是绝对分数。

如何理解 GSPO / DAPO

前者更强调 sequence-level 优化的稳定性，后者更强调大规模 reasoning RL 训练中的实操可复现性与系统细节。

Agent

Agent 训练要素：为什么它比普通回答式 RL 更难

Agentic RL 的难点不在“会不会求梯度”，而在于环境更复杂。状态不仅是用户问题， 还包括工具输出、历史上下文、执行日志、观察结果和外部世界变化。

状态 State

当前对话、已执行动作、工具返回、文件系统快照、网页内容、测试结果，都可能进入状态表示。

动作 Action

不仅是输出 token，也可能是调用搜索、执行 shell、编辑文件、发起 API、结束任务等结构化动作。

奖励 Reward

可以是最终任务成功率，也可以混合答案正确性、格式合法性、调用成本、步数惩罚、测试通过率、verifier 分数。

信用分配 Credit Assignment

当最后一步成功时，真正关键的也许是前面某次搜索、某次回退或某次工具选择。长轨迹信用分配是 agent RL 的核心难题。

很多 agent 训练可抽象成一个 POMDP：

$$o_t \rightarrow h_t \rightarrow a_t \rightarrow o_{t+1}, r_t$$
$o_t$ 是观察，$h_t$ 是历史或隐状态，$a_t$ 是动作。大模型经常通过上下文窗口近似地维护这个历史。

如果是多步推理或工具使用场景，你还需要同时关心下面几件事：

  1. 探索是否足够。采样太保守，模型学不到新策略；采样太激进，训练信号又会噪声很大。
  2. 奖励是否可验证。能自动判分的任务最适合做大规模 RL，比如数学、代码、结构化问答。
  3. 轨迹是否太长。上下文越长，训练成本越高，梯度越容易受到无关 token 干扰。
  4. 工具观察是否稳定。真实 API 波动、网页变化、执行环境差异都会让奖励分布漂移。
Practice

训练与评测实践：把理论落到系统设计上

真正的瓶颈通常不只在算法，还在 rollout、验证器、并发采样、环境控制和评测指标体系。 很多实验结果不稳定，根源不是 loss 公式，而是训练系统没有把轨迹质量和奖励质量管住。

冷启动

先用高质量 SFT 给模型打一个可执行的行为先验，否则 RL 阶段会把大量预算浪费在无意义探索上。

奖励系统

优先选择可自动验证的奖励。数学看答案，代码看测试，搜索看命中率，agent 任务看任务完成度与轨迹合法性。

Rollout 系统

准备好并发采样、工具沙箱、超时控制、日志记录和轨迹回放能力。很多训练瓶颈其实不在模型，而在 rollout 基建。

评测体系

不能只看平均 reward。还要看 pass@k、步骤长度、工具调用成功率、重复率、格式错误率、越狱率和成本。

一个典型的 Agentic RL 闭环可以总结为：

$$\text{Prompt} \rightarrow \text{Sample Trajectories} \rightarrow \text{Verifier / Reward} \rightarrow \text{Advantage} \rightarrow \text{Policy Update}$$

如果你自己搭实验，最建议盯住这几类指标：

  • 任务指标：最终正确率、pass@1、pass@k、任务完成率。
  • 过程指标：平均步数、平均输出长度、工具调用次数、重复搜索率、空转率。
  • 训练指标：KL 漂移、reward 均值/方差、有效样本比例、loss spike 次数。
  • 成本指标：每条轨迹 token 数、每轮 rollout 耗时、验证器成本、GPU 利用率。
场景 奖励最好怎么设计 为什么
数学推理 最终答案正确 + 过程格式约束 + 长度轻惩罚 自动判分容易，适合大规模探索
代码生成 / SWE Agent 测试通过率 + patch 可执行性 + 超时惩罚 结果验证直接，轨迹长但反馈明确
搜索 / 浏览器 Agent 目标命中 + 引用质量 + 非法动作惩罚 需要兼顾结果正确和过程合规
通用聊天 偏好模型 + 安全规则 + 人工抽检 真值难定义，更接近传统 RLHF
Papers

代表论文：把 Agent RL 读成一张图

这一页把“Agentic RL 热潮”拆成几组代表论文。你可以把它们理解成三条主线： 先是 ReAct / ToT / SWE-agent 把 agent 交互问题定义清楚，再是 DeepSeekMath / DeepSeek-R1 / DAPO / GSPO 把 reasoning RL 推热， 最后是 ReST-MCTS*、rStar-Math、Agent Q 这类工作把搜索、轨迹合成和训练闭环接起来。

A. Agent 形态与任务接口
2022 Reason + Act
ReAct: Synergizing Reasoning and Acting in Language Models

ReAct 把“推理 token”和“环境动作”交错在同一条轨迹里，是现代 Agent 轨迹格式和 action-observation 循环的重要起点。

2023 Search
Tree of Thoughts: Deliberate Problem Solving with Large Language Models

ToT 把推理过程从单链条扩展为树搜索，是后续 MCTS、best-of-n、deliberate inference 和 search-based agent 方法的关键前导工作。

2024 Software Agent
SWE-agent: Agent-Computer Interfaces Enable Automated Software Engineering

SWE-agent 让模型在代码仓库、测试命令和文件编辑环境里执行长轨迹任务，直接推动了“面向真实环境的 agent 训练”讨论。

2024 GRPO 起点
DeepSeekMath: Pushing the Limits of Mathematical Reasoning in Open Language Models

DeepSeekMath 是公开语境里理解 GRPO 的重要入口，它把组相对奖励与数学 reasoning 训练直接连接起来。

B. DeepSeek-R1 之后的 reasoning RL 热潮
2025 Reasoning RL
DeepSeek-R1: Incentivizing Reasoning Capability in LLMs via Reinforcement Learning

DeepSeek-R1 让“先冷启动再强化学习”的 reasoning 路线被更多人看见，也是你理解 2025 年社区为何集体转向 RL reasoning 的代表论文。

2025 System + Algorithm
DAPO: An Open-Source LLM Reinforcement Learning System at Scale

DAPO 重点不是只讲一个 loss，而是把 decoupled clipping、dynamic sampling 和大规模 RL 系统工程一起开源，适合从“怎么真跑起来”角度读。

2025 Sequence-Level RL
Group Sequence Policy Optimization

GSPO 代表一种序列级更新视角：直接基于 sequence likelihood 和 sequence clipping 来优化，更贴近 sequence-level reward 的粒度。

2024 Offline Step-Level RL
Improving Multi-Step Reasoning Abilities of Large Language Models with Direct Advantage Policy Optimization

DAPO 这篇方法论文强调 step-level offline RL，用 critic 产生更密的优势信号，适合和大规模系统版 DAPO 一起对照阅读。

C. MCTS 合成成功轨迹与搜索驱动自训练
2024 MCTS + Self-Training
ReST-MCTS*: LLM Self-Training via Process Reward Guided Tree Search

这篇非常贴近你说的“MCTS 合成成功轨迹”。它用 process reward 引导树搜索，收集更高质量的 reasoning traces，并把这些轨迹继续喂回模型做自训练。

2025 Verified Trajectories
rStar-Math: Small LLMs Can Master Math Reasoning with Self-Evolved Deep Thinking

rStar-Math 使用 MCTS rollout 生成逐步验证过的 reasoning trajectories，再用这些轨迹训练策略模型和过程偏好模型，是“搜索合成成功样本”的代表工作。

2024 MCTS + Agent Learning
Agent Q: Advanced Reasoning and Learning for Autonomous AI Agents

Agent Q 把 guided MCTS、self-critique 和基于 agent interaction 的偏好优化结合起来，说明搜索不仅能做 test-time enhancement，也能反过来产生训练数据。

怎么读这组论文

Pitfalls

常见误区：很多实验不是算法输，而是系统没搭对

误区 1：reward 高了就等于能力强了

如果奖励设计得太窄，模型会 reward hack。比如学会钻 verifier 漏洞，而不是真会推理。

误区 2：只看最终答案，不看轨迹质量

Agent 场景里，成功背后可能伴随大量无效搜索、重复工具调用和高成本路径，这会直接影响线上可用性。

误区 3：把缩写当创新本身

看到新方法时，第一件事不是背全称，而是问它到底修改了奖励、优势、裁剪、采样还是归一化。

误区 4：忽略长度偏置

长推理场景里，sequence-level reward 容易偏向更长或更短回答，必须检查是否需要长度归一化、token weighting 或 decoupled clipping。

误区 5：没有强基线就直接上 RL

如果 SFT、best-of-n、test-time search 都没跑稳，直接上 RL 往往只会把问题放大，不会神奇解决一切。

误区 6：忽略数据与环境漂移

Agent 往往连接真实环境。网页结构变了、测试集变了、API 限流了，奖励分布就会变化，必须监控数据管线。

Roadmap

学习路线：从入门到能读论文、能搭实验

如果你的目标是把 Agentic RL 真正学会，而不是只背术语，最有效的方法是按“数学基础 → 对齐方法 → 推理 RL → agent 环境 → 搜索与轨迹合成”去推进。

阶段 1：打地基

先学马尔可夫决策过程、策略梯度、baseline、KL、PPO。目标是看懂“为什么要 clip，为什么要 regularize”。

阶段 2：理解 LLM 对齐

把 SFT、RM、PPO、DPO 的关系弄清楚，知道什么时候是偏好学习，什么时候是真正的在线 RL。

阶段 3：进入 reasoning RL

重点理解组采样、相对优势、序列级奖励、verifier、拒绝采样与 best-of-n 的关系。

阶段 4：进入 agent RL

关注工具调用、多步轨迹、环境状态、长上下文、信用分配和 rollout 基础设施。

一个很实用的阅读顺序是：

  1. 先看 PPO 和 REINFORCE，把公式真正写明白。
  2. 再看 RLHF / DPO，弄懂“偏好优化”和“在线策略优化”差在哪。
  3. 然后看 ReAct、ToT、SWE-agent，理解 agent 到底把问题复杂化在什么地方。
  4. 最后回来看 GRPO、GSPO、DAPO、ReST-MCTS*、Agent Q，你会发现它们大多是在为“长推理、多样采样、稳定更新”服务。
$$\text{真正的能力增长} \approx \text{强先验模型} + \text{可验证奖励} + \text{稳定 rollout} + \text{合适的策略更新}$$

如果把这页教材压缩成一句话，那就是： Agentic RL 不是某个缩写的胜利，而是把大模型从“会说”推向“会做、会试、会修正”的系统工程。

继续扩展这份教材时，可以沿三条线往下挖：一条是“公式线”，深入到 GAE、importance sampling、variance reduction； 一条是“系统线”，深入到 rollout engine、verifier pipeline、tool sandbox； 一条是“论文线”，沿 ReAct → ToT → ReST-MCTS* → rStar-Math → Agent Q 追踪“搜索生成轨迹如何反哺训练”。
