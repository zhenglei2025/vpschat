Formulation

形式化问题定义

Agentic RL 的第一步不是选算法，而是把问题建模清楚。LLM 生成文本可以被形式化为一个 MDP； 当 agent 需要与外部环境交互时，问题变成 POMDP。在此之上，RLHF、Reasoning RL、Agentic RL 三种范式在 state/action/reward/horizon 上有本质差异，这些差异直接决定了算法选型和训练策略。

1.1 Token-level MDP：LLM 生成作为序贯决策

一个自回归 LLM 的生成过程可以形式化为一个确定性转移的 MDP $\mathcal{M} = (\mathcal{S}, \mathcal{A}, P, R, \gamma)$：

  • 状态空间 $\mathcal{S}$：所有可能的 token 前缀序列。$s_t = (x, a_1, a_2, \dots, a_{t-1})$，其中 $x$ 是 prompt。
  • 动作空间 $\mathcal{A} = \mathcal{V}$（词表），每步输出一个 token。
  • 转移函数 $P(s_{t+1} | s_t, a_t)$：确定性拼接——$s_{t+1} = s_t \oplus a_t$。这与 Atari/Robotics 的 stochastic transition 本质不同。
  • 策略 $\pi_\theta(a_t | s_t)$：即 LLM 在给定前缀下的 next-token distribution。
  • 奖励 $R$：通常只在序列结束时给出（sparse），$r_T = R(x, y)$，中间步 $r_t = 0$。
  • 折扣因子：在大多数 LLM RL 设定中 $\gamma = 1$（undiscounted episodic 问题）。
为什么 deterministic transition 很重要？ 在经典 RL 中，环境的随机性是学习困难的主要来源之一。但在 token-level LLM MDP 中，给定 $s_t$ 和 $a_t$，$s_{t+1}$ 是完全确定的（字符串拼接）。 所有随机性来自 策略本身 $\pi_\theta(a_t|s_t)$，而不是环境。这意味着：(1) 不需要 model-based 方法来学环境模型； (2) return 的方差主要来自策略采样而非环境噪声；(3) Monte Carlo return 估计是无偏的，且不受 transition noise 影响。
$$\text{Trajectory: } \tau = (s_0, a_1, s_1, a_2, \dots, s_{T-1}, a_T, s_T)$$ $$P(\tau | \pi_\theta) = \prod_{t=1}^{T} \pi_\theta(a_t | s_t) \cdot \underbrace{P(s_{t} | s_{t-1}, a_t)}_{=1 \text{ (deterministic)}}$$
由于转移是确定的，轨迹概率完全由策略决定：$P(\tau | \pi_\theta) = \prod_{t=1}^{T} \pi_\theta(a_t | s_t)$。
1.2 Turn-level POMDP：Agent 与环境的交互建模

当 LLM 需要调用工具、执行命令、浏览网页时，问题从 token-MDP 上升到 turn-level POMDP。 此时存在两层嵌套的决策结构：

$$\text{外层（turn-level）: } o_1 \to a_1^{\text{turn}} \to o_2 \to a_2^{\text{turn}} \to \cdots$$ $$\text{内层（token-level）: } a_k^{\text{turn}} = (w_1, w_2, \dots, w_n) \sim \prod_{i=1}^n \pi_\theta(w_i | s_i)$$
外层的每个 action $a_k^{\text{turn}}$ 本身是一个多 token 的序列，由内层 token-level 策略生成。观察 $o_k$ 包含环境返回（工具输出、测试结果、网页内容等）。

partial observability 的来源有三个：

  1. 上下文窗口截断：历史可能超过上下文长度限制，导致早期信息丢失。
  2. 环境状态不可见：agent 只看到工具返回的 text，看不到底层系统状态（如内存使用、并发进程等）。
  3. 信息不完整：搜索引擎只返回 snippet，文件系统只显示 ls 结果，API 返回可能被截断。
两层结构的训练困难 如果在 token-level 做 RL 更新，每个 token 都需要 advantage signal，但 reward 通常只在 turn-level 或 trajectory-level 给出。 如果在 turn-level 做 RL 更新，则需要把每个 turn（整段文本）视为一个 macro-action，这会丢失 token 粒度的 credit assignment。 这个 粒度不匹配 是 Agentic RL 的核心技术挑战之一。
1.3 三范式形式化对比

RLHF、Reasoning RL、Agentic RL 看似都是 "用 RL 训练 LLM"，但建模层面存在本质差异：

RLHFReasoning RLAgentic RL
State $s = [\text{prompt}]$ $s = [\text{prompt} \oplus \text{CoT prefix}]$ $s = [\text{prompt} \oplus \text{history} \oplus \text{observations}]$
Action 完整回答 $y$ CoT + final answer 结构化动作（text / tool call / terminate）
Transition Trivial（单轮） Deterministic（self-extend） Stochastic（环境响应不确定）
Reward $r = R_\phi(x, y)$（偏好模型） $r = \mathbb{1}[\text{answer correct}]$ $r = f(\text{task}, \text{trajectory quality})$
Horizon $T \approx$ 回答长度 $T \approx$ CoT 长度 $T \approx$ 多轮交互长度（可能数千 token）
核心难点 Reward model 质量 推理链 credit assignment 环境随机性 + 长程 credit assignment
代表工作 InstructGPT, Llama-2-Chat DeepSeek-R1, GRPO Agent Q, RAGEN, Search-R1
1.4 Non-stationarity 分析

经典 RL 理论假设环境是 stationary 的（转移和奖励函数不随时间变化），但 Agentic RL 面临多层 non-stationarity：

环境非平稳

网页内容随时间变化、API 接口升级、搜索引擎结果波动。agent 今天执行同一动作序列，得到的 observation 可能和昨天不同，导致 reward 分布漂移。

策略非平稳

On-policy 采样要求 rollout 策略和更新策略一致，但异步训练中 learner 更新了 $k$ 步后，rollout worker 仍在用旧策略采样，造成 importance ratio 偏离。

Reward Model 非平稳

如果使用 learned reward model（而非 rule-based verifier），随着策略分布变化，reward model 面对 out-of-distribution 的输入，给出的分数可能失准。

数据分布非平稳

训练过程中 prompt 的难度分布可能发生 curriculum 变化，或者因为过滤策略的改变（如 DAPO 的 dynamic sampling）导致训练数据分布持续漂移。

从理论上说，Policy Gradient Theorem 的成立条件之一是 reward function 和 transition function 在训练过程中不变。 当这个假设被违反时，梯度估计可能是 biased 的。实践中的缓解策略包括： (1) 使用 rule-based verifier 替代 learned reward model 以减少 reward drift； (2) 保持 on-policy 采样以减少 policy staleness； (3) 使用沙箱隔离环境以保证 transition 一致性。
