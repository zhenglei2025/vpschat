训练系统：从 Rollout 架构到 Failure Mode 诊断

"我的 GRPO reward 不涨"——80% 的原因不在 loss 公式，而在训练系统。 Rollout 延迟、on-policy staleness、显存不足、advantage normalization 的 micro-batch 实现 bug， 都可能让一个正确的算法跑不出结果。

7.1 Rollout 系统架构

LLM RL 训练的瓶颈通常在 rollout（采样）而非 gradient update。 核心架构决策是 Actor-Learner 分离的程度：

同步架构（TRL 默认）

同一组 GPU 交替进行 rollout 和 training。
Pro：实现简单，rollout 策略和训练策略完全同步（on-policy）。
Con：GPU 利用率低——rollout 时不训练、训练时不 rollout。

异步架构（veRL, OpenRLHF）

Rollout workers 和 Learner workers 分离，通过 Ray 调度。
Pro：吞吐量高，GPU 利用率好。
Con：引入 policy staleness——rollout 用 $\pi_{\theta_k}$ 采样，learner 已更新到 $\pi_{\theta_{k+n}}$，importance ratio 偏离。

On-policy Staleness 的量化 设 rollout 用 $\pi_{\theta_k}$ 采样，learner 在 $n$ 次更新后变为 $\pi_{\theta_{k+n}}$。 此时 importance ratio 的偏离程度 $\approx \exp(n \cdot \eta \cdot \|\nabla\mathcal{L}\|)$（一阶近似）， 其中 $\eta$ 是学习率。当 $n$ 较大或 $\eta$ 较大时，clipping 机制可能不足以纠正这个偏离。 实践中通常控制 $n \leq 1$（即每轮 rollout 后只做 1-4 epoch 更新）。

7.4 分布式训练的 Subtle Issues
  • Advantage Normalization 在 Micro-batch 中的 Bug： 如果在 micro-batch 内做 advantage normalization（减均值除标准差）， 不同 micro-batch 的 normalization 统计量不同，导致梯度不一致。 正确做法是在全局 batch 上计算 normalization 统计量，然后分发到各 micro-batch。
  • Reference Model 的显存管理： KL 计算需要 $\pi_{\text{ref}}$，但额外加载一个 full-size model 很贵。 常见优化：(1) 共享 backbone 只冻结 ref model 的部分层；(2) 每隔 $N$ 步才更新 ref model；(3) 将 ref model offload 到 CPU/NVMe。
  • vLLM Rollout 与 Training 的切换开销： 使用 vLLM 加速 rollout 时，需要在 continuous batching 推理模式和 training 模式之间切换， 涉及权重同步和 GPU 内存重分配。veRL 的 weight synchronization protocol 是解决这个问题的关键设计。
